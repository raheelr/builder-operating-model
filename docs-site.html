Start Here

You don't need to know how to code. You just need to know what you want to build.

The only two things you need to know before starting

1. Your first command: /builder help I want to build [describe your idea]
Describe what you want in plain English. Builder will take it from there.

2. Your rescue command: /builder help [whatever you're confused about]
Use this any time you're stuck, lost, or don't know what to do next.

Everything else in this documentation exists when you want to go deeper. You don't need to read it before you start. Builder will surface what you need, when you need it.

Your First 30 Minutes — Step by Step

1
Open Claude Code and type your first command

Start a new conversation. Type this — fill in your own idea:

/builder help I want to build [describe your idea here]

For example:

  • /builder help I want to build an app to track my school's book loans
  • /builder help I want to build a website for my family's small business
  • /builder help I want to build a tool that tracks my soccer team's season stats
  • /builder help I want to build something that lets our club sign up for events

Your idea doesn't need to be perfectly worded. Just describe what it does.

2
Read Builder's response

Builder will respond with:

  • What it understood about your idea — read this carefully and correct anything that's wrong
  • What to do next, in plain English
  • The exact command to type next

If it misunderstood something, just say "no, that's not right — what I meant was..." and keep going.

3
Type the next command it gives you

Builder's response ends with the exact command to run next. This is usually /builder init. Type it.

Builder will ask you questions — one at a time. It won't present a form or a list of 10 things to answer all at once. Each turn is one question. Answer it, and Builder asks the next one. Answer in plain, conversational English:

  • "I want people to log in with their email address"
  • "It needs to work on phones too"
  • "Only about 20–30 people will use this at first"
  • "I don't need it to accept payments"
  • "I have no idea — what do you recommend?"

That last one is always a valid answer. Builder will suggest the sensible default and explain why.

4
When asked about mode, say "business leader" or "non-technical"

At some point Builder asks: "Who is the primary builder — a developer, or a business or domain leader directing the build?"

Answer: "business leader" or "non-technical."

This activates plain-language mode for everything going forward. Builder will never use technical jargon with you, and will always explain things in terms of your project — not in terms of code.

5
Review the summary Builder shows you — then approve it

Builder will show you a summary of everything it understood: what you're building, who it's for, what the major pieces are, and what gets built first.

This is your last chance to fix anything before building starts. Read it carefully. Then say:

  • "Yes, that looks right" — if it's correct
  • "No, [this part] is wrong — what I meant was..." — if anything needs adjusting

Once you approve, you're set up. Building starts after this.

You're now set up

The hard part is done. Every session from here starts with you saying what you want — in plain English — and Builder figures out what to do. You approve the work before it's considered done. You never need to read or write code.

What Each Session Looks Like From Here

After setup, every session is one of three things:

🔨 Building
Say: "What should we build next?" or "Keep going."

Builder picks up where it left off and builds the next piece of your project.
✅ Checking
Say: "Is this ready? Can we show someone?"

Builder reviews the work and reports back in plain language. You decide if it's done.
🔧 Fixing
Say: "Something's broken" or "This isn't working."

Builder diagnoses the problem and fixes it. You don't need to know what went wrong.

You direct which one happens. Builder does the technical work. You approve before anything is considered done.

When You Get Confused

Type this — any time, in any session:

/builder help [describe what you're confused about]

Examples:

  • /builder help what just happened and what should I do next?
  • /builder help what does "spec" mean?
  • /builder help we haven't touched this in two weeks — where were we?
  • /builder help how do I show someone what we've built so far?
  • /builder help something broke and I don't know what
  • /builder help I feel like we're going in circles — what's the problem?

Builder will explain the situation in plain English, tell you exactly what to do next, and give you the command.

What you will NOT need to do
  • Write or read any code
  • Understand technical terms like "database," "API," or "deployment" — Builder explains anything you need to know when the time comes
  • Read the rest of this documentation before starting — it's here when you want to go deeper, but you don't need it now
  • Figure out what to build next — Builder tracks what's planned and always knows the next step

Builder Operating Model

A structured methodology for AI-native software development. A human builder orchestrates specialized AI agents through explicit modes, shared state docs, and disciplined adaptive guardrails — producing production-grade software, not vibe code.

The Core Idea

You bring the vision. The operating model brings the structure. Claude reads your codebase and intent, derives the right modules, agents, and roadmap — then you orchestrate through seven canonical modes. The guardrails compound quality over time.

The 5-Stage AI Maturity Framework

Most teams are at Stage 3 — using AI for code completion. The Builder takes you to Stage 4: structured agent orchestration with human judgment at the helm.

The Harness Philosophy

The Harness is the Product

"The harness is the product. The model is just the engine." The value isn't in the AI model — it's in the layer of rules, evidence requirements, self-improvement loops, and architectural constraints that encode your project's knowledge into AI behavior. Models change. The guardrails compound.

What Makes This Different from "Vibe Coding"

  • Vibe coding has no architecture doc. This model starts with one.
  • Vibe coding has no quality gate. This model reviews every diff.
  • Vibe coding has no idea backlog. This model gates every feature through IDEAS.md.
  • Vibe coding has no state tracking. This model maintains CURRENT_STATE.md as the honest record.
  • Vibe coding hopes the AI gets it right. This model assumes it won't and builds review into the process.

Foundation

Getting started with the Builder — from a blank slate, from Scout Team discovery, or upgrading an existing project.

Project Init

/builder init

Every project starts here. Init begins by asking a single question:

How would you like to start?

1. I know what I'm building — We'll capture your intent, derive the structure, and scaffold the project. ~20 min

2. I have a problem space — let's reason through it first — We'll think through the problem together, explore approaches, make decisions, and then scaffold from that understanding. 1-2 hrs

Fast path: I know what I'm building

For builders with clarity on what they want. The bootstrapper will:

  1. Read existing code — scan structure, languages, frameworks, patterns
  2. Understand intent — ask what's being built, who it's for, the stack
  3. Derive canonical modules — propose named scopes from code + intent
  4. Derive subagents — group modules by ownership boundaries, create agent definitions
  5. Derive initial roadmap — 3-5 phases; Phase 1 is always "prove the core loop once"
  6. Generate scaffold — write all shared state docs
  7. Confirm with builder — present everything for review before committing

Deep path: Let's reason first

For builders who have a problem space but haven't decided what to build, what approach to take, or what the scope should be. The reasoning is the work. The scaffold is a byproduct.

1. Problem Framing
What's the problem? Who experiences it? What happens without a solution? What does success look like? Claude doesn't just capture answers — it probes, challenges assumptions, surfaces implications.
2. Constraints & Unknowns
Timeline, team, budget, compliance. What do you know vs. what are you guessing? Claude pushes on the unknowns — which ones matter most? Which need resolution before building?
3. Requirements Reasoning
Claude reasons WITH you. "You said multi-tenancy — let me think about what that implies for data, auth, deployment, and compliance." Edge cases explored. Dependencies surfaced.
4. Approach Exploration
2-3 possible technical approaches with trade-offs. Claude presents pros, cons, risks, and a recommendation with reasoning. Not "what's your stack?" but "here's why approach A fits better."
5. Decisions
The builder decides. Tech stack, v1 scope, what's deferred. Each decision captured as an ADR — not just what, but the full reasoning that led there.
6. Scaffold
Same output as fast path, but docs carry full reasoning context. ARCHITECTURE.md has a "Context & Reasoning" section. First ADRs already exist. Roadmap grounded in explored trade-offs.
Why the deep path matters

The reasoning that produces the answers IS the guardrails being built. Skip it, and you're building the wrong thing faster. A scaffold without reasoning is a guess. A scaffold grounded in structured problem exploration is a foundation. Every agent reads this context at session start. Every decision can be traced back to the conversation that produced it.

Advanced Elicitation

After reasoning through a decision, stress-test it using named elicitation methods. Claude offers methods after initial reasoning. The builder picks one. Especially powerful during deep init, ADR creation, and architecture discussions.

Pre-mortem
"Assume this failed — what went wrong?" Surfaces risks that optimism obscures.
First Principles
"What do we actually know?" Strips away assumptions to find bedrock facts.
Red Team vs. Blue Team

How It Works in Practice

You run /builder reason or /builder adr. Claude reasons through the question, presents a recommendation. Then: "Want to stress-test this? I can run a pre-mortem, first-principles analysis, red-team/blue-team, inversion, Socratic questioning, or constraint removal." You pick one. Claude runs it. The decision either survives the stress test or you find the gap before it becomes a production problem.

The Agent Model

N specialized subagents + one builder (the human). The builder orchestrates through modes and shared state — there is no LLM orchestrator above the specialists.

The Builder Role

The Builder IS

  • An architect who makes judgment calls about what to build, in what order
  • A quality gate who reviews every piece of work before merge
  • An orchestrator who spawns agents, manages parallel work, resolves conflicts
  • A domain expert who validates the system solves the actual problem
  • A delegator of routine quality checks to automated review agents

The Builder is NOT

  • Not a developer who writes code — agents do that
  • Not a PM who tracks hours — the model tracks capability state
  • Not optional — without human judgment, agents solve the wrong problem
  • Not a scout — doesn't do discovery, but must consume what scouts produce

Canonical Modules

Every project defines its canonical modules — coherent scopes of the project's surface area.

Properties of a good module:

  • Large enough to be a planning target (≥3 related features)
  • Small enough to be internally consistent (one owner, one purpose)
  • Has code anchors (files or folders you can point to)
  • Has exactly one owner (a subagent or the builder)

Modules are the only valid scopes for planning sessions. The module list lives in docs/ARCHITECTURE.md.

Self-Configuring Agents

During init, Claude proposes agents from your code + intent. The sweet spot is 3-5 agents — more agents = more coordination overhead and drift risk. Only add more when the project has genuinely non-overlapping domains.

Parallel Execution

When agents work on non-overlapping areas, they run in isolated git worktrees — each gets its own copy of the repo. When complete:

  1. Worktree has a clean branch with commits
  2. Builder reviews the diff
  3. Builder merges to main
  4. Worktree is cleaned up
Hard Rule

Agents never force-push, rebase main, or alter history. Only the builder handles merges.

Bounded Autonomy

Not all tasks carry the same risk. Bounded autonomy assigns a risk tier to each task and scales the review process accordingly. Low-risk work flows fast. High-risk work gets maximum scrutiny. Architecture decisions stay with the builder entirely.

TierScopeExamplesAgent Authority
Auto-safeZero-risk changesDocumentation, test additions, refactors with no behavior change, formattingExecute and commit. Builder reviews later in batch. No pre-approval needed.
StandardNormal feature workNew endpoints, UI components, business logic, integrationsExecute in worktree. Builder reviews before merge. The default tier.
High-scrutinySensitive domainsAuthentication, payment processing, PII handling, data migrations, encryptionExecute in worktree. Adversarial review required. Multi-model review recommended.
Builder-onlyStructural decisionsArchitecture changes, module boundary shifts, tech stack decisions, security model changesNot delegated to agents. The builder makes these decisions through reason, architecture, or ADR modes.
Why This Matters

Without bounded autonomy, every task gets the same review weight. Builders waste time reviewing documentation fixes with the same rigor as auth code. With it, safe work flows at full speed while sensitive work gets the scrutiny it deserves. The tiers are assigned during planning and can be overridden at advance time.

Signal-First Output Format

Every Builder response follows a three-part structure regardless of mode or audience:

1. Signal
3 bullets max. Plain language. What matters right now and what needs a decision. Always first, before any detail. The builder reads this and knows the state in 10 seconds.
2. Detail
Full reasoning, findings, or output. Follows the signal. In enterprise mode, collapsed by default — the builder says "tell me more about X" to expand.
3. What's next?
1–3 immediately actionable next steps, specific to what just happened. Never generic. The builder responds with "yes" or the phrase shown and Builder executes. Non-negotiable — closes every response.
Proactive Guidance — Never Silent

After EVERY mode — advance, review, status, planning, triage, debug, correct-course, architecture, security, adr, sync, feedback, and all discussion commands — Builder appends a "What's next?" section. The builder never has to ask "what do I do now?" The first suggestion is always the single most logical next step given what just happened. Suggestions are specific to the current project state, never generic ("you can run any mode").

The Planning Session (5 Activities)

A planning session on a single module follows five activities in order:

  1. Load module context — read CURRENT_STATE, ROADMAP, IDEAS for the module. Flag any doc drift.
  2. Backlog grooming — walk every task and IDEAS entry. Still relevant? Description accurate? Phase right? Re-sequence, demote, split, merge.
  3. New feature and idea capture — brainstorm what's missing. Formal IDEAS entries for non-trivial ideas, parked in memo for minor ones.
  4. Dependency mapping — upstream dependencies, downstream dependents, broken chains.
  5. Cross-module prioritization — does this module's plan fit the system's overall plan, or is it only coherent in isolation?

TDD Mode (Opt-In)

An optional modifier for advance sessions that enforces the Red-Green-Refactor cycle. The agent writes the test first, verifies it fails, then writes the minimum code to pass.

Mode: advance. TDD mode. Phase 2, task P2.3.
Red — Write failing test → Green — Minimum code to pass → Refactor — Clean up, tests still pass → Commit

Each cycle is atomic: write the test, verify failure, write the code, verify pass, refactor, verify pass, commit. The agent repeats the cycle for each behavior in the task spec.

When to Use TDD Mode

  • Business logic with clear inputs/outputs
  • API endpoints with defined contracts
  • Calculations, transformations, validations
  • Bug fixes (write failing test that reproduces the bug first)

When to Skip It

  • UI layout and styling work
  • Early prototyping / spikes
  • Infrastructure and config setup
  • One-off scripts and migrations
Why This Matters

TDD mode means the business logic is provably correct before the builder ever reviews it. The test exists as a specification: given these inputs, produce these outputs. The code is written to satisfy that specification. No ambiguity, no "it should work" — the test proves it does.

Quality & Verification

Six quality gates, evidence-based review, automated testing, and the explicit contrast with vibe coding.

The 6 Quality Gates

  1. Tests exist and pass — unit, integration, or e2e matching the scope of change
  2. Linting is clean — no new warnings
  3. No hardcoded secrets — env vars, config in .gitignore, or secrets manager
  4. CURRENT_STATE.md updated — in the same commit as the code
  5. Interlock deliverable still works — regression check after merge
  6. Security scan passes — if configured

Reconciliation Step

Review checks quality — "Is the code good?" Reconciliation checks completeness — "Did we build everything the spec says?" These are different concerns. A diff can pass review with flying colors and still be missing half the acceptance criteria.

After a task is reviewed and before it merges, reconciliation walks the spec's acceptance criteria one by one and marks each as implemented or not:

Reconciliation Report — P2.1 JWT Auth + RBAC

AC-1  JWT access token issued on login           ✓ IMPLEMENTED
AC-2  Refresh token rotation on use               ✓ IMPLEMENTED
AC-3  RBAC decorator for admin/broker/viewer       ✓ IMPLEMENTED
AC-4  Token expiry: access 15min, refresh 7d       ✓ IMPLEMENTED
AC-5  Rate limiting on /auth/token                 ✗ NOT IMPLEMENTED
AC-6  Audit log entry on failed login attempts     ✗ NOT IMPLEMENTED

Result: 4/6 acceptance criteria met. 2 gaps identified.
Review vs. Reconciliation

Review asks: "Is what you built correct and well-crafted?" Reconciliation asks: "Did you build everything the spec says?" A task that passes review but fails reconciliation has gaps. A task that passes reconciliation but fails review has quality issues. Both must pass before merge.

Spec-Driven Independent Testing

The testing agent reads the spec, not the code. It writes tests from acceptance criteria without ever seeing the implementation. If the tests diverge from what the code actually does, that divergence is a bug — either the spec is wrong or the implementation is.

This is the principle: separate the builder from the validator. The agent that built the feature has confirmation bias — it "knows" the code works because it wrote it. A separate agent that only reads the spec has no such bias. It writes tests for what should exist, not what does exist.

Spec (acceptance criteria) → Testing agent writes tests → Tests run against implementation → Divergence = bug
Why Spec-Driven Matters

When a building agent writes its own tests, it tests what it built. When a testing agent writes tests from the spec, it tests what should have been built. The gap between those two is where bugs hide.

Two-Stage Review

Every review runs two distinct passes, in order. Separating spec compliance from code quality prevents a common failure mode: accepting well-written code that doesn't match what was asked for, or approving correct-but-messy code because it technically works.

Pass 1: Spec Compliance

"Did you build what was asked?"

  • Compare deliverable against task spec
  • Check acceptance criteria met
  • Verify scope — no extras, no missing
  • Approach aligns with architecture + ADRs

Pass 2: Code Quality

"Is it built well?"

  • File-pattern rules followed
  • Tests exist and cover changes
  • No hardcoded secrets
  • Error handling present
  • CURRENT_STATE updated
Why Two Passes

When spec compliance and code quality are evaluated simultaneously, reviewers unconsciously trade one off against the other. Clean code that solves the wrong problem gets approved. Correct code with security gaps gets through because "it works." Two passes force both questions to be answered independently — each must pass on its own merits.

Protocol Enforcement Gates

Four hard gates enforce quality at the process level. These are not suggestions — they are structural constraints that prevent bad work from flowing forward.

Build-Verify Gate
Tests must pass before review. Failed tests block the diff from being shown to the builder. No "it mostly works" — green or nothing.
State-Doc Gate
CURRENT_STATE.md must update alongside code changes. If the code changed but the state doc didn't, the commit is incomplete.
Review-Before-Merge Gate
No auto-merge. Every piece of work passes through review mode. The builder approves explicitly — nothing merges silently.
IDEAS Gate
No feature work outside the roadmap. New ideas go to IDEAS.md for evaluation during triage — not into the current task as a "quick addition."
Default: Enforcement, Not Trust

The default is enforcement, not trust. The builder can override any gate with explicit justification — but skipping silently is not an option. If you bypass a gate, you say why. The system records the override. This keeps velocity high while maintaining accountability.

Hooks as Enforcement

Gates are procedural — the methodology says "do this before that." Hooks are mechanical — the system physically prevents violations. The difference between a sign that says "wash your hands" and a door that won't open until you do.

Three hook types enforce quality at the tooling level:

Hook TypeWhen It RunsWhat It EnforcesExample
Pre-commitBefore every commitCode-level constraints that must never be violatedLint passes, no secrets in diff, no console.log in production code, CURRENT_STATE.md modified if source files changed
Pre-mergeBefore worktree merges to mainIntegration constraints between the branch and the mainlineAll tests pass on merged code, no merge conflicts, state docs consistent across branches, reconciliation report generated
Post-taskAfter agent marks task completeCompleteness and evidence requirementsEvidence artifacts exist (test output, screenshots), checkpoint file written, CURRENT_STATE.md updated with new capabilities
Procedural vs. Mechanical Enforcement

A procedural gate says "you should run tests before merging." A hook runs the tests and blocks the merge if they fail. Procedural enforcement depends on discipline. Mechanical enforcement depends on tooling. The Builder uses both — but when a constraint is critical, it becomes a hook, not a suggestion.

Adaptive Guardrails

The layer between the AI model and your codebase that encodes project knowledge into AI behavior. It compounds in value over every session.

How It Works

Builder corrects Claude → Detect pattern → Propose rule → Builder approves → Permanent improvement

After each session where the builder corrects Claude:

  1. Detect the correction pattern — what category of mistake was it?
  2. Propose a new rule — file-pattern rule or architecture principle that prevents recurrence
  3. Builder approves or skips — no automatic rule creation
  4. Track improvement — session 1 had X corrections, session 10 had Y
  5. The system gets permanently better — corrections become rules, rules prevent future corrections

Compounding Quality

This is the compounding differentiator. The guardrails don't just prevent known mistakes — they learn from new ones.

SessionState
Session 1Rough. ~5 starter rules from bootstrapper. Builder corrects frequently.
Session 10Smoother. 12-15 rules. Corrections less frequent. Common patterns are caught automatically.

Shared Guardrail Library

Guardrails compound within a project — but the Shared Guardrail Library compounds them across the organization's portfolio. Organizations maintain a shared repository of stack-specific guardrail modules. When a project uses Sitecore, .NET, Databricks, or any known technology, the builder inherits battle-tested rules from previous projects using that stack.

Guardrails compound not just within a project, but across the organization's portfolio. A rule learned on one Sitecore project prevents the same mistake on every future Sitecore project. The first project using a stack pays the learning cost. Every subsequent project inherits the result.

The Definition Path

The path from "I have a big idea" to "this is fully understood, documented, and ready to build."

Builder has many commands. The Definition Path tells you which ones to run, in what order, and what "done" looks like at each step before moving to the next. Without this path, builders jump from idea to code and agents guess at what was meant. With it, every component is agreed upon before anyone starts building.

The Key Principle

Specs are the bridge. Without a spec, the gap between "big idea" and "working code" is filled by agents guessing what you meant. A spec makes the agreement explicit — before anyone starts building — so that review is a verification of a shared understanding, not a negotiation after the fact.

The Six Steps

Step 1 — Understand the problem
A structured 1–2 hour conversation that reasons through the actual business problem before anything is designed. What's the pain? Who experiences it? What does success look like? What are you guessing vs. what do you know?

Command: /builder init (deep path)
Done when: Architecture is grounded in the real problem. First decisions are recorded. Everyone agrees on what's being built and why.
Step 2 — Map the system
Name every major component of the system. Give each one a status (not started, designed, being built, live), an owner, and a pointer to the decisions that govern it. Define the rule for keeping this map current.

Command: /builder system-map
Done when: You can describe the whole system in one page. A new team member knows the major pieces, their status, and who owns each one.
Step 3 — Spec each component
For each component entering development: write a clear spec before a line of code is written. What does it do? What does it explicitly NOT cover? Who uses it? What are the requirements? What are the acceptance criteria — the objective tests that prove it works?

Command: /builder spec <component>
Done when: Every component planned for the next phase has a spec. Acceptance criteria are written. Open questions have owners.
Step 4 — Validate coherence
Verify that all the docs agree with each other. Architecture principles, roadmap tasks, specs, security requirements — do they tell a consistent story? Are there contradictions? Are dependencies correctly ordered?

Command: /builder ready-check
Done when: Green readiness report. No contradictions. Safe to build.
Step 5 — Build
Agents execute against the specs. They know exactly what they're building, what "done" looks like, and what they must NOT build (out-of-scope items are explicit in the spec). No guessing.

Command: /builder advance
Done when: Task complete, tests pass, current state updated.
Step 6 — Verify against the spec
Confirm that what was built matches what was agreed. The review panel checks quality. Reconciliation checks completeness — every acceptance criterion from the spec is verified.

Command: /builder review + reconciliation
Done when: Reconciliation passes. All acceptance criteria verified. Code merged.

When to Run It

Starting a new phaseSteps 2–4 (re-map, re-spec new components, re-validate)
A component enters development for the first timeSteps 3–4 minimum (spec it, validate coherence)

Progressive Disclosure — One Thing at a Time

The most common failure in enterprise mode isn't the wrong vocabulary — it's too much information at once. Even in plain language, a wall of text stops a non-technical builder cold.

Enterprise mode is a conversation, not a report. Builder applies these rules automatically:

One question per turn
During setup and planning, Builder asks one question at a time — then waits. The builder sets the pace. Never a form, always a conversation.
One next step, not a menu
The closing suggestion is one option: the single most obvious next action. "Say 'yes' to continue." Not three paths requiring a decision.
Documents stay behind a summary
When Builder writes a spec, roadmap, or architecture doc, it shows a 3-bullet summary — not the document. "Here's what it covers. Ready to keep going?" The builder asks "show me" if they want details.
Commentary is off by default
Builder doesn't explain why something matters unless asked. It asks the next question or states what was done. The builder says "why?" if they want the reasoning.
Pause-and-check, not options
After every significant step: "Does that look right? Say 'yes' to continue or tell me what to change." Not a menu of alternatives.
~150 word cap per response
If a response is approaching 150 words, compress. What's the one thing the builder needs to know right now? Say that. Everything else waits until asked.

The Scenario

You're building PolicyHub — a multi-tenant SaaS platform for insurance brokers to manage policy applications, quotes, and renewals. It has a React frontend, a Python/FastAPI backend, a PostgreSQL database, and integrates with three external carrier APIs. You have the idea. You have an empty directory. Let's go.

Phase 0: Inception 1-2 hours

You start with one command

/builder init

The Builder model reads your directory (empty in this case) and asks how you want to start. You don't know exactly what to build yet — you have a problem space, some ideas, and constraints. You choose: "I have a problem space — let's reason through it first."

The reasoning session begins

Problem framing

Claude doesn't ask "what are you building?" It asks "what problem exists?"

You explain: insurance brokers spend 3+ hours per policy quote because they have to log into each carrier's portal separately, re-enter the same client data, wait for each quote, then manually compare results in a spreadsheet. It's slow, error-prone, and brokers hate it.

Claude probes: "Walk me through what those 3 hours look like. Where does the broker start? What data do they enter first? When they get a quote back, what do they do with it?" You describe the workflow. Claude surfaces an insight: "So the bottleneck isn't just the data entry — it's that each carrier has a different data format, and the broker is essentially doing human-powered API translation. Is that right?"

Yes. That's the real problem. Not "we need a portal" — we need to eliminate the human translation layer between brokers and carriers.

Constraints & unknowns

Claude asks about constraints. You lay them out:

  • Timeline: Pilot with 3 broker firms in 3 months
  • Team: Just you as builder, AI agents do the implementation
  • Compliance: Insurance data has PII requirements, SOC 2 needed for enterprise brokers
  • Budget: Cloud-hosted, need to keep infrastructure costs under $500/mo at pilot scale
  • Biggest unknown: Carrier APIs vary wildly. Some have REST APIs, some have SOAP, one still uses EDI files. You don't know how hard the integrations will be.

Claude flags the unknown: "The carrier API variability is your highest risk. If we design the architecture wrong, each integration becomes a custom project. If we get the adapter pattern right, each new carrier is just a configuration exercise. This decision should drive the architecture, not follow it."

Requirements reasoning

Claude reasons through each requirement with you, not just records it:

"You said multi-tenancy — each brokerage firm is a tenant. Let me think about what that implies:

  • Data model: tenant_id on every table, or schema-per-tenant? Schema-per-tenant gives you stronger data isolation for SOC 2, but complicates migrations.
  • Auth: JWT must carry tenant context. Every API endpoint needs tenant scoping. A broker at Firm A must never see Firm B's data.
  • Carrier credentials: each firm has their own API keys for each carrier. Those are secrets per-tenant, not per-app.
  • The credential management alone is a significant piece of infrastructure. Should that be Phase 1 or can quotes work with shared test credentials initially?"

You decide: shared test credentials for Phase 1, per-tenant credentials in Phase 2. Claude captures the rationale.

Approach exploration

Claude presents three architectures:

ApproachProsCons
A: Monolith + adapter patternSimple deployment, fast to build, adapter interface isolates carrier complexityAll carriers in one process, harder to scale independently
B: MicroservicesEach carrier is its own service, independent scalingOverkill at pilot scale, complex deployment, team of one
C: Monolith now, extract laterFast start with clean boundaries so extraction is easy when neededRequires discipline to maintain boundaries

Claude recommends C: "Given your 3-month timeline, solo team, and the unknown carrier API complexity — start monolith, but design the adapter layer with clean interfaces so any carrier can be extracted into its own service later. The Builder Operating Model's module boundaries will enforce the separation discipline."

You agree. This becomes ADR-001.

Decisions crystallize

After 45 minutes of reasoning, the decisions have emerged organically — not from a form, but from structured thinking about the actual problem:

  • Tech stack: React + TypeScript, FastAPI, PostgreSQL with schema-per-tenant, Redis for caching carrier responses

  • Database: PostgreSQL for schema isolation compliance (ADR-002)
  • V1 scope: 3 carriers, quote comparison, basic policy management. Deferred: renewals, claims, document management
  • Phase 1: One carrier with test credentials, prove the adapter pattern works end-to-end

Architecture & Sources

Introduction

The Builder Operating Model didn't emerge in a vacuum. It synthesizes the strongest patterns from the autonomous coding landscape — tested across real tools, validated by production use, and integrated into a system where every piece reinforces every other piece.

Over the past year, the industry has converged on a set of hard-won lessons about how AI agents should build software. Specifications must precede code. Reviews must be independent. Guardrails must be enforced mechanically, not on the honor system. Context windows must be managed deliberately. Testing must be driven by the spec, not the implementation.

Different teams discovered different subsets of these truths. CodevOS built a multi-model review pipeline. Superpowers enforced TDD discipline and two-stage verification. The Builder Operating Model draws from all of them. But it isn't a feature matrix or a best-of compilation. It's an integrated system where reasoning feeds execution, execution feeds verification, verification feeds the guardrails, and the guardrails feed back into everything.

Why It Hangs Together

Five Principles of Integration

1. Every capability has a ROLE. It exists to solve a specific problem in the pipeline. There are no "nice to have" features bolted on for completeness.

2. Capabilities COMPOSE. Multi-model review + adversarial review + two-stage review + checkpoint preview aren't four separate tools — they're four layers of the same review process.

3. The builder is the integration point. All capabilities flow through a human making judgment calls. The builder doesn't need to understand every agent's internals — they evaluate outputs through structured presentations.

4. The guardrails are the learning loop. Corrections anywhere in the system become permanent improvements everywhere. This is how the system gets smarter, not just per-project but per-organization.

5. Shared state docs are the coordination mechanism. Agents don't talk to each other. They talk through docs — ROADMAP.md, CURRENT_STATE.md, checkpoints, ADRs. This makes the system inspectable and debuggable by the builder at any point.
