Section 1

Introduction

The Builder Operating Model didn't emerge in a vacuum. It synthesizes the strongest patterns from the autonomous coding landscape — tested across real tools, validated by production use, and integrated into a system where every piece reinforces every other piece.

Over the past year, the industry has converged on a set of hard-won lessons about how AI agents should build software. Specifications must precede code. Reviews must be independent. Guardrails must be enforced mechanically, not on the honor system. Context windows must be managed deliberately. Testing must be driven by the spec, not the implementation.

Different teams discovered different subsets of these truths. CodevOS built a multi-model review pipeline. Superpowers enforced TDD discipline and two-stage verification. BMAD developed adversarial review and structured upstream methodology. The PE AI Way introduced independent testing and reconciliation. Carl Tierney articulated the philosophy that "the harness is the product — the model is just the engine."

The Builder Operating Model draws from all of them. But it isn't a feature matrix or a best-of compilation. It's an integrated system where reasoning feeds execution, execution feeds verification, verification feeds the guardrails, and the guardrails feed back into everything.

Why This Paper Exists

This paper maps each capability to its source, explains how it was adapted for the Builder, and demonstrates why the whole is greater than the sum of its parts. It's written for the PE team — the people who will use this methodology every day to build production systems for clients.

Why It Hangs Together

Five Principles of Integration

1. Every capability has a ROLE. It exists to solve a specific problem in the pipeline. There are no "nice to have" features bolted on for completeness.

2. Capabilities COMPOSE. Multi-model review + adversarial review + two-stage review + checkpoint preview aren't four separate tools — they're four layers of the same review process.

3. The builder is the integration point. All capabilities flow through a human making judgment calls. The builder doesn't need to understand every agent's internals — they evaluate outputs through structured presentations.

4. The guardrails are the learning loop. Corrections anywhere in the system become permanent improvements everywhere. This is how the system gets smarter, not just per-project but per-organization.

5. Shared state docs are the coordination mechanism. Agents don't talk to each other. They talk through docs — ROADMAP.md, CURRENT_STATE.md, checkpoints, ADRs. This makes the system inspectable and debuggable by the builder at any point.

Section 4

How It Works Holistically — A Production System Walkthrough

Feature lists describe what a system has. This section shows what a system does. We walk through a real engagement — building a production system — showing how all the capabilities work together. Not feature-by-feature, but as an integrated system.

Week 0
Foundation

/builder init (deep path) — 2 hours of structured reasoning with the client's domain expert. The builder doesn't ask "what do you want?" — the system guides the conversation through problem exploration, constraint mapping, stakeholder identification, and architecture reasoning.

Advanced elicitation: Pre-mortem on the architecture — "Assume we fail in 6 months. What went wrong?" Three ADRs emerge from the reasoning session, documenting decisions before a line of code exists.

/builder guardrails inherit dotnet-core,azure-services,hipaa-compliance — 35 stack-specific rules inherited from the organizational guardrail library.

/builder ready-check — Green. Architecture, roadmap, security, guardrails all coherent. The system confirms the project is ready to build.

Meanwhile, the Scout Team runs its first discovery session in parallel — 10 agents analyzing source documents, extracting workflows, surfacing contradictions.

Week 1
First Build Cycle

/builder sync — absorb scout findings. Narrative briefing read and understood. Two new requirements surfaced from discovery are integrated into the roadmap.

/builder advance — 3 tasks spawned in parallel worktrees. Bounded autonomy classifies each: 1 auto-safe (docs setup), 2 standard (API endpoint, business logic).

The business logic agent uses TDD mode for quote calculation. Red-Green-Refactor cycle produces 12 tests, each demonstrating the Red phase before implementation.

Agent completes → hooks enforce automatically: linter runs (PostToolUse), CURRENT_STATE.md check (Stop hook), scope validation (PreToolUse prevented one out-of-scope file edit).

Build-verify gate: all tests pass → review unlocked.

Multi-model review: fresh agent reviews independently (never saw the build session).

Two-stage review: Pass 1 checks spec compliance ✓. Pass 2 checks code quality: 1 finding — missing input validation on a currency field.

Builder reviews with checkpoint preview: comprehension-ordered walkthrough shows orientation first, then changes by concern, with risk tags on the auth code.

Merge. Self-improvement proposes a guardrail rule from the input validation correction. Builder approves. Guardrail rule #8 created.

Week 2
The System Compounds

Guardrails: 12 rules (6 inherited + 4 from init + 2 from corrections). Each rule prevents future instances of a past mistake.

Planning session surfaces a dependency between the notification module and the user preferences module that the builder had missed. The 5-activity structure catches it: load context, groom tasks, capture new work, map dependencies, cross-module check.

Scout feedback: 2 implementation gaps sent back to discovery. The bridge is bidirectional — building informs discovery, not just the other way around.

Phase 1 complete → reconciliation: 4/5 acceptance criteria met, 1 deferred to Phase 2. The reconciliation caught that AC-3 was partially implemented — the review had marked it as complete.

Ready for testing checklist surfaces. Builder tests functionally against the spec. Evidence collected: test output, API response screenshots, database query results.

Retrospective captures 3 lessons. One becomes a guardrail rule.

Week 4
Scale

3 agents running in parallel. Bounded autonomy handles the routine. The builder focuses on the 2 high-scrutiny tasks — payment processing and PII handling — with adversarial review enabled. Zero findings on the payment module requires written justification from the reviewer.

Guardrails: 18 rules. Corrections are rare now — agents produce code matching the project's conventions on first attempt. The self-improvement loop is working.

A requirement changes mid-build. /builder correct-course — architecture revalidated, roadmap adjusted, readiness re-checked, ADR written for the change rationale. Continue.

An IDEAS gate catches a drive-by feature request from a stakeholder. The idea is captured in IDEAS.md for evaluation rather than immediately entering the roadmap.

Week 8
Production

Guardrails: 24 rules (6 inherited stack rules + 18 project-specific). Each one a permanent lesson that makes every future session better.

ADRs: 11 — every significant decision documented with full reasoning. New team members can understand not just what was built, but why.

The testing agent has verified 187 acceptance criteria against specs, not code. It never saw the implementation — it tested from the specification.

Reconciliation confirms: 100% of specified acceptance criteria implemented. Zero scope drift. The reconciliation report is the proof.

Security discussion revisited → audit passed. The pre-mortem from Week 0 prevented the architecture mistake that would have caused the audit to fail.

/builder guardrails contribute multi-tenant-isolation — the best project rule proposed back to the shared library for future projects across the organization.

The Compounding Effect

Week 1's input validation correction became a permanent guardrail rule. The inherited stack guardrails prevented 35 convention violations that never even happened. The reconciliation caught a missing acceptance criterion the review didn't. The multi-model review caught a security issue the building agent missed. The advanced elicitation in Week 0 prevented an architecture mistake that would have cost 2 weeks of rework.

No single capability does this. The SYSTEM does this.

Section 5

What's Original to the Builder

Not everything was borrowed. The Builder Operating Model introduced several capabilities that don't exist in any of the source frameworks. These are the innovations that emerged from applying agent-assisted development to real client engagements.

Two-Mode Init

No other framework offers structured problem reasoning as an alternative to "tell me what to build." The deep init path guides exploration before specification.

IDEAS Gate

Explicit idea-to-roadmap evaluation. All new features go through IDEAS.md before entering the roadmap. No drive-by features.

Reshape

Evolving module boundaries without losing in-flight work. Tasks are remapped, not deleted. Understanding deepens; the system adapts.

Planning Sessions (5-Activity)

Structured module-level backlog grooming: load, groom, capture, map dependencies, cross-module check. Prevents drift and surfaces hidden dependencies.

Scout Team Bridge

Bidirectional discovery-to-build loop. Working software on Day 1. Building informs discovery; discovery informs building.

Correct Course

Mid-build requirements revalidation with architecture coherence check. Requirements change; the system handles it gracefully.

Builder Modes (8 Canonical)

Eight named session types with explicit contracts: advance, status, triage, planning, review, delta, retrospective, reason. Each mode has a defined purpose and output.

Self-Improvement Visibility

Metrics on the guardrails' effectiveness over time. Track how many corrections become rules, how many violations the rules prevent, and where the system is still learning.
